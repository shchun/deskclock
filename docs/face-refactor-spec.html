02무엇이 아픈가

중복 near-duplicate 페이스 쌍

face1↔face5(Minimal), face3↔face7(Typography), face4↔face8(Neon)은 레이아웃이 사실상 동일하고 색/그림자만 다르다. 그런데 각자 완전히 분리된 scoped CSS + DOM + tick 블록을 가진다.

스멜 tick() 스위치보드

같은 시간 포맷팅을 face마다 textContent로 손수 반복(963~997줄). face를 추가할 때마다 이 함수에 블록을 덧붙여야 한다.

스멜 수동 ref 선언 30+

const f4t=el('f4-time'),… 식으로 일일이 잡는다. id 규칙(f{N}-time)이 암묵적이라 자동 수집 가능한데도 손으로 나열.

스멜 fitActive() 매직 스트링 분기

if(id==='face3'||id==='face7')…, face11 특례가 함수 안에 하드코딩. face 추가 시 여기도 수정 필요.

스멜 매초 innerHTML 재생성

face4·8·9는 매 tick마다 innerHTML=h12+'<span…>'로 콜론 span을 통째로 다시 만든다 → 깜빡임 애니메이션 위상 리셋·불필요한 파싱.

문서 CLAUDE.md 표류

구조 설명(face 수·줄 수·FACES 표)이 실제와 어긋나 신규 작업 시 오해 유발. 리팩터링과 함께 정정.

03제안 — 데이터 주도 Face 레지스트리

핵심 아이디어: 각 face가 자기 갱신 방법을 자기 설정에 들고 있게 한다. tick()은 공통 시간 모델 하나를 만들고 활성 face(혹은 전체)의 render(parts)만 호출. ref/fit/render가 모두 FACES 한 곳에 모인다.

            ┌─────────────────────────────────────────────┐
   tick() → │  parts = { h12, hh, mm, ss, ampm,           │   공통 시간 모델 1개
            │            dateLong, dateShort, h24 }        │   (face별 반복 제거)
            └───────────────────┬─────────────────────────┘
                                │  for face of FACES: face.render(parts, refs)
        ┌───────────────────────┼───────────────────────┐
        ▼                       ▼                       ▼
  FACES['face4'] = {      FACES['face6'] = {      FACES['face11'] = {
    name, theme,            name, theme,            name, theme,
    fit:{mode:'wrap',…},    fit:{mode:'wrap'},      fit:{mode:'goo',…},
    render(p,r){ … }        render: flip,           render(p,r){ blobs }
  }                       }                       }
        └─── refs 자동 수집: querySelectorAll('[data-bind]') ───┘
// 변경 전 — tick() 안에 face4 갱신이 하드코딩
f4t.innerHTML = h12 + '<span class="f4-colon">:</span>' + mm;  // 매초 재파싱
f4a.textContent = ampm; f4s.textContent = ss; …

// 변경 후 — 콜론 span은 1회 생성, 숫자만 갱신 + 설정에 위치
render(p, r){ r.h.textContent = p.h12; r.m.textContent = p.mm;
             r.s.textContent = p.ss; r.a.textContent = p.ampm; }
04접근 방식 선택

"얼마나 깊이 갈 것인가"의 결정. 셋은 배타적이지 않고 A ⊂ B이며, C(중복 쌍 병합)는 A/B 위에 얹는 별도 축이다.

A. 최소 정리 추천 출발점

노력 S · 위험 낮음
  • tick() → 공통 parts 모델 + face별 render()
  • ref 자동 수집(data-bind)
  • fitActive를 face.fit 설정으로 이동
  • 동작·픽셀 결과 불변, 되돌리기 쉬움
  • 중복 CSS는 그대로 남음

B. 템플릿화

노력 M · 위험 중간
  • A 전부 포함
  • face 마크업을 <template>로, JS가 인스턴스화
  • HTML 본문이 크게 짧아짐
  • DOM 생성 타이밍·폰트 로드 순서 재검증 필요
  • 디버깅 시 한 단계 우회

C. 중복 쌍 병합

노력 M · 위험 중상 (외형 영향)
  • 3쌍 → 3 face + data-variant 토큰화
  • CSS 최대 ~40% 감소
  • 디자인 결정: 10개 유지 vs 6+토글로 축소?

05마이그레이션 단계 (확정 스코프 — A + 부가 3건)
1
공통 시간 모델 추출tick() 상단에서 parts 객체(h12,hh,mm,ss,ampm,dateLong,dateShort,h24) 1회 계산.
2
ref 자동 수집각 갱신 노드에 data-bind="time|sec|ampm|date" 부여 → face 진입 시 querySelectorAll로 수집. 수동 const 30개 제거.
3
face별 render(parts, refs)face마다 갱신 로직을 FACES 항목으로 이동. flip(face6)·goo(face11)는 기존 전용 함수를 render로 래핑.
4
colon span 1회 생성face4·8·9의 매초 innerHTML 제거 → 마크업에 콜론 span 고정, 숫자 노드만 textContent 갱신.
5
fit 설정화fitActive의 id 분기를 face.fit={mode,selector,maxScale,avail}로 대체.
6
localStorage id 기반 전환 [확정]deskclock.face를 배열 인덱스 → face id로 저장. 읽을 때 id→인덱스 매핑, 기존 숫자값은 호환 폴백 처리. 향후 B/C에서 순서 변경에도 부팅 face 안전.
7
검증 + 문서 [확정]npx cap copy android 후 실기기 10 face 순회 확인. CLAUDE.md face 수·줄 수·FACES 표 갱신.
06리스크 & 실패 모드
localStorage 인덱스 호환 [해소]: deskclock.face는 FACES 배열 인덱스를 저장한다. → 5단계에서 id 기반 저장으로 전환(숫자값 폴백 포함)하기로 확정. 향후 B/C의 순서 변경에도 안전.
flip/goo 특수 렌더: face6(animationend 구동)·face11(블롭 nth-child 애니메이션)은 단순 textContent 모델에 안 맞음 → render 인터페이스가 이 둘을 1급으로 수용해야 함.
폰트 로드 타이밍: 템플릿화(B)에서 DOM 생성이 늦어지면 document.fonts.ready 기반 fit 재계산 순서가 어긋날 수 있음.
스코프 경계: sonar PoC(우측하단 SONAR 링크·하단 sonar script)는 "검증 후 삭제" 표시된 임시 코드. 이번 리팩터링 범위에 포함하지 않음 — 별도로 처리.
